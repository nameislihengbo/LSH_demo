## 零、本规则的三层架构 / Three-Layer Structure of This Rule

本规则本身是三层架构的体现：

| 层级 | 内容 | 对应章节 |
|------|------|----------|
| **Structure 结构层** | 跨语言结构定义、数据类型映射规范 | 第一节~二节 |
| **Rule 规则层** | FFI 一致性规则、序列化规则、错误处理规则 | 第三~四节 |
| **Execution 执行层** | 测试验证、内嵌测试示例 | 第五节：测试规范 |

**代码即架构**：代码文件本身是 LSH 架构的实体化。Rust 将测试内嵌于代码中，正是 Execution 层的体现。

## 一、跨语言结构定义 / Cross-Language Structure Definitions

### 1.1 结构对应原则

跨语言结构定义必须保持**语义一致性**：

| Rust 结构 | Python 结构 | 说明 |
|-----------|-------------|------|
| `struct Element` | `dict` / `Element` 类 | 核心元素结构 |
| `ElementId` | `str` | 元素唯一标识 |
| `Vec<T>` | `list[T]` | 数组类型 |
| `HashMap<K, V>` | `dict` | 键值对 |
| `Option<T>` | `T \| None` | 可选类型 |

### 1.2 结构定义模板

```rust
// Rust: 结构定义 (lsh-protocol/rust/src/element/element.rs)
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Element {
    pub id: ElementId,
    pub name: String,
    pub category: Category,
    pub tags: Vec<String>,
    pub properties: HashMap<String, Value>,
    pub position: [f64; 3],
    pub size: [f64; 3],
    pub rotation: [f64; 3],
    pub visible: bool,
}
```


### 1.3 类型映射表

| Rust 类型 | Python 类型 | JSON 类型 | 说明 |
|-----------|-------------|-----------|------|
| `bool` | `bool` | `boolean` | 布尔值 |
| `i32` / `i64` | `int` | `integer` | 整数 |
| `f32` / `f64` | `float` | `number` | 浮点数 |
| `char` / `String` | `str` | `string` | 字符串 |
| `Vec<T>` | `list[T]` | `array` | 数组 |

| `Option<T>` | `T \| None` | `null` | 可空 |

## 二、FFI 一致性规则 / FFI Consistency Rules

### 2.1 FFI 导出规范

Rust 导出到 Python 必须通过 PyO3，并遵循以下规范：

```rust
// Rust FFI 导出 (lsh-protocol/rust/src/ffi/python.rs)
use pyo3::prelude::*;
use pyo3::wrap_pyfunction;

#[pyclass]
pub struct PyElement {
    inner: Element,
}

#[pymethods]
impl PyElement {
    #[new]
    fn new(id: String, name: String, category: String) -> Self {
        PyElement {
            inner: Element::new(id, name, category),
        }
    }

    fn to_dict(&self) -> PyResult<HashMap<String, PyObject>> {
        Ok(self.inner.to_dict())
    }
}

#[pyfunction]
fn create_element(id: String, name: String, category: String) -> PyResult<PyElement> {
    Ok(PyElement::new(id, name, category))
}

pub fn register_module(m: &PyModule) -> PyResult<()> {
    m.add_class::<PyElement>()?;
    m.add_function(wrap_pyfunction!(create_element, m)?)?;
    Ok(())
}
```

## 六、代码变更工作流 / Code Change Workflow

### 6.1 Structure-Rule-Execution 工作流

代码变更必须按照 LSH 三层架构处理，无需更改的层跳过：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    代码变更工作流                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. Structure (结构层) - 是否需要改结构？                             │
│     ├── 需要：修改数据结构、类型定义、接口签名                        │
│     └── 不需要：跳过 ✅                                              │
│                                                                     │
│  2. Rule (规则层) - 是否需要改逻辑？                                  │
│     ├── 需要：修改业务规则、约束条件、处理流程                        │
│     └── 不需要：跳过 ✅                                              │
│                                                                     │
│  3. Execution (执行层) - 是否需要改验证？                             │
│     ├── 需要：修改测试、基准、检查清单                                │
│     └── 不需要：跳过 ✅                                              │
│                                                                     │
│  脏标记：每层修改后标记 dirty，确保下游感知变更                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 6.2 脏标记机制

```rust
// Element.version 就是脏标记
// 每次修改自增，下游可检测变更
pub fn bump_version(&mut self) {
    self.version = self.version.wrapping_add(1);
    self.updated_at = Utc::now();
}
```

| 操作 | Structure 脏 | Rule 脏 | Execution 脏 |
|------|-------------|---------|-------------|
| 修改数据结构 | ✅ | ✅ (联动) | ✅ (联动) |
| 修改业务逻辑 | - | ✅ | ✅ (联动) |
| 修改测试 | - | - | ✅ |
| 仅修改注释 | - | - | - |

**核心原则**：Structure 变更联动 Rule 和 Execution，Rule 变更联动 Execution，Execution 变更独立。

### 6.3 变更检查清单

- [ ] Structure 层是否需要修改？（数据结构、类型定义）
- [ ] Rule 层是否需要修改？（业务逻辑、约束条件）
- [ ] Execution 层是否需要修改？（测试、验证）
- [ ] 脏标记已更新（version bump）
- [ ] 联动层已同步修改

## 七、验证规则 / Validation Rules

### 7.1 验证即执行

**核心原则**：代码验证是 Execution 层的实体化。测试不是附加物，而是代码存在的必要条件。

```
Structure (定义) + Rule (逻辑) → Execution (验证)
```

### 8.2 LSH ID 统一格式

**唯一抽象源**：[lsh_format.md §2.2](../../lsh-protocol/docs/lsh_format.md#L56-L95)

**格式**：`{prefix}[_group...]-{timestamp}-{hash}`

**核心原则**：三段皆语义，三段皆可插拔。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    LSH ID 三段可插拔语义                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  {语义:属性}-{语义:时间戳}-{语义:哈希}                                │
│                                                                     │
│  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐ │
│  │   属性段          │  │   时间戳段        │  │   哈希段          │ │
│  │ prefix[_group...] │  │ timestamp        │  │ hash             │ │
│  │ 外部: - 内部: _   │  │ 外部: - 内部: _  │  │ 外部: - 内部: _  │ │
│  └──────────────────┘  └──────────────────┘  └──────────────────┘ │
│                                                                     │
│  分隔符规则：外部用 -，内部用 _（反之亦可，但默认外部 -）             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 8.3 分隔符规则

**核心原则**：组内分隔只能用与外部不同的分隔符。

| 层级 | 默认分隔符 | 说明 | 示例 |
|------|-----------|------|------|
| 段间（外部） | `-` | 分隔属性/时间戳/哈希 | `code_rust-20260511_090002-e4f5g6` |
| 段内（内部） | `_` | 分隔 prefix/group | `code_rust` |
| 时间戳内部 | `_` | 分隔日期/时间 | `20260405_143052` |
| 哈希内部 | 无 | 连续字符 | `a3f8c2` |

**规则**：外部 `-` 则内部 `_`，反之亦然。默认外部用 `-`。

### 8.4 三段可插拔语义

每段都是语义，每段都可插拔：

| 段 | 语义 | 可插拔形式 | 示例 |
|----|------|-----------|------|
| **属性段** | prefix[_group...] | `device`, `code_rust`, `theory_gradient_paradigm` | 1+N 模式 |
| **时间戳段** | 时间语义 | `20260405`, `20260405_143052`, `1778489606` | 日期/日期时间/Unix |
| **哈希段** | 唯一语义 | `a3f8c2`, `19970`, `1a2b3c4d5e6f` | 默认6位，可增强 |

**时间戳可插拔**：

| 格式 | 示例 | 精度 | 适用场景 |
|------|------|------|----------|
| 日期 | `20260405` | 天 | 文件命名、规则 |
| 日期时间 | `20260405_143052` | 秒 | 精确追溯 |
| Unix 时间戳 | `1778489606` | 秒 | 训练 checkpoint |

**哈希可插拔**：

| 格式 | 示例 | 位数 | 适用场景 |
|------|------|------|----------|
| 短哈希 | `a3f8` | 4 | 低冲突场景 |
| 默认哈希 | `a3f8c2` | 6 | 通用 |
| 增强哈希 | `1a2b3c4d5e6f` | 12 | 高唯一性要求 |

### 8.5 默认规则与简化规则

**默认规则**（完整格式）：

```
{prefix}[_group...]-{timestamp}-{hash}
```

| 规则 | 格式 | 示例 | 适用场景 |
|------|------|------|----------|
| **默认** | `prefix_group-timestamp-hash` | `code_rust-20260511_090002-e4f5g6` | 正式环境 |
| **属性简化** | `prefix-timestamp-hash` | `code-20260511-e4f5g6` | 无 group |
| **时间戳简化** | `prefix_group-date-hash` | `code_rust-20260511-e4f5g6` | 日期精度 |
| **哈希简化** | `prefix_group-timestamp-h4` | `code_rust-20260511-e4f5` | 低冲突场景 |
| **极简** | `prefix-date-h4` | `code-20260511-e4f5` | 快速原型 |

**平衡原则**：属性语义 > 时间精度 > 哈希位数。属性不可省略，时间戳和哈希可按需简化。

### 8.6 快捷搜索规则

**首字母缩写**：用各段首字母快速定位。

| 完整 ID | 缩写 | 规则 |
|---------|------|------|
| `structure_latex_tikz-20260511_110002-x1y2z3` | `sl_l_t-2026-x1` | 段首字母 + `_` 分隔 |
| `code_rust-20260511_090002-e4f5g6` | `cr-2026-e4` | prefix 首字母 |
| `theory_gradient_paradigm-20260511_120004-f7h6i5` | `tg_p-2026-f7` | prefix_group 首字母 |

**缩写规则**：

```
{prefix首字母}[_group首字母...]-{timestamp前4位}-{hash前2位}
```

| 缩写部分 | 规则 | 示例 |
|---------|------|------|
| 属性缩写 | 各段首字母，`_` 连接 | `sl_l_t` ← structure_latex_tikz |
| 时间戳缩写 | 年份4位 | `2026` ← 20260511 |
| 哈希缩写 | 前2位 | `x1` ← x1y2z3 |

**局部特征校验**：缩写不要求完整，支持部分匹配。

| 搜索 | 匹配 | 说明 |
|------|------|------|
| `sl-2026-x1` | `structure_*-2026*-x1*` | 属性+时间+哈希 |
| `sl-*-x1` | `structure_*-*-x1*` | 仅属性+哈希 |
| `*-2026-x1` | `*-2026*-x1*` | 仅时间+哈希 |
| `*-*-x1y2z3` | `*-*-x1y2z3` | 精确哈希（最精确） |

**双重哈希**：哈希本身是第一重，首字母缩写是第二重。精确搜索用完整哈希，模糊搜索用缩写。
